# Plan / Roadmap Best Practices

## 基本原則

- plan は目的、範囲、優先順位、依存関係、検証可能な中間成果を可視化する
- roadmap は時間表ではなく、価値提供の順序と判断材料を示す
- phase は完了後に検査できる中間成果で区切る
- task は実行者が迷わず着手でき、レビュー可能な成果物を残せる粒度にする
- plan は固定計画ではなく、検証結果に応じて更新される正本として扱う

## phase 分割

- 1 phase は 1 つの検証可能なゴールを持つ
- phase の完了条件は観測可能な状態で書く
- phase 間の依存関係を明示する
- 後続 phase の前提になる検証を省略しない
- 大きすぎる phase は、探索、基盤、移行、検証、リリース準備などの観点で分ける

## task 分割

- 1 task は AI エージェントが 1 セッションで完了しやすい作業量にする
- 1 task は 1 Markdown ファイルにする
- task は「何を作るか」だけでなく「どう検証するか」まで含める
- 複数の独立した成果物、別々の検証方法、別々のリスクを持つ作業は分割する
- 調査、設計、実装、移行、検証は混ぜすぎない
- task の粒度が大きい場合は、完了条件を満たすために必要なファイルやテストが多すぎないか確認する

## task に必ず書くこと

- 目的
- 前提条件
- 作業内容
- 完了条件
- 検証方法
- 依存関係
- 成果物

## 避けること

- plan task に AI の作業ログを書く
- `.agents/tasks/` の文脈キャッシュを plan の正本として扱う
- 検証できない「対応する」「整える」「改善する」だけの task を置く
- 複数 phase にまたがる task を 1 ファイルにまとめる
- 完了条件と検証方法がない task を置く
