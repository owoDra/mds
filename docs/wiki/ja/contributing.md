# コントリビューション

このページでは、mds に関する報告、提案、ドキュメント改善を行うときの基本方針を説明します。

このリポジトリのソースコード自体は、mds の実装 Markdown から生成されているわけではありません。`mds/*` と `editors/vscode` 配下の checked-in source / test を直接編集し、mds でこの repository 自体を再生成する前提は置きません。

## 歓迎する貢献

mds では、次のような貢献を歓迎します。

- 不具合報告
- 再現手順の追加
- ドキュメント改善
- 仕様の分かりにくい部分の指摘
- 利用例の提案
- エラーメッセージや診断内容に関する改善提案
- 対応してほしい言語や配布方法に関する要望

## 提案や報告で確認すること

提案や報告を行うときは、次の点を確認してください。

| 内容 | 確認すること |
| --- | --- |
| 不具合報告 | 再現手順、実行したコマンド、期待した結果、実際の結果を確認します。 |
| 仕様の提案 | 何を解決したいのか、どの利用場面で必要なのかを確認します。 |
| ドキュメント改善 | 利用者が次に何をすればよいか分かる説明になっているか確認します。 |
| 利用例の追加 | その例で必要なファイル、コマンド、期待結果を確認します。 |

## 推奨する検査

ソースコードを変更する場合は、通常の Rust、JavaScript、Python のプロジェクトとして検査します。

Rust の変更では、可能な範囲で次の検査を実行してください。

```bash
cargo fmt --all --check
cargo check --workspace
cargo test --workspace
cargo clippy --workspace --all-targets -- -D warnings
```

VSCode 拡張を変更した場合は、追加で次を実行してください。

```bash
cd editors/vscode
npm install
npm run compile
```

mds を利用するサンプルパッケージを用意している場合は、次の smoke check も行えます。

```bash
cargo run -p mds-cli -- check --package examples/minimal-ts
cargo run -p mds-cli -- build --package examples/minimal-ts --dry-run
```

## ドキュメントを書くときの注意

読み手が前提を知らなくても理解できるように、説明を省略しすぎないでください。

略語や専門用語を使う場合は、最初に意味を説明してください。

公開ドキュメントには、mds というツールの利用方法、設計、仕様、運用上の注意を書きます。特定の開発環境だけで使う作業メモや、公開利用者に関係しない情報は含めません。

## 不具合報告に含めるとよい情報

不具合を報告する場合は、次の情報があると原因を調べやすくなります。

- 実行したコマンド
- 期待した結果
- 実際の結果
- 終了コード
- 対象の `mds.config.toml`
- 対象の実装 Markdown の最小例
- 利用している Rust、Node.js、Python のバージョン
