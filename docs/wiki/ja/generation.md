# 生成の仕組み

このページでは、mds が authoring-v2 の Markdown 文書から package output を作る流れを説明します。

## 何が出力を生むか

- source doc では `Source` fence が executable output の元になります。
- test doc では `Test` fence が executable output の元になります。
- prose、見出し、overview doc 自体はそれだけでは code を生成しません。

## logical module id

mds は `.mds/source` または `.mds/test` の内側の path から logical module id を導出します。

例:

| Markdown doc | logical module id |
| --- | --- |
| `.mds/source/greet.ts.md` | `greet` |
| `.mds/source/http/client.ts.md` | `http/client` |
| `.mds/test/greet.ts.md` | `greet` |

この module id は output pattern と wiki-style link の両方で使われます。

## 既定の output planning

既定設定では:

```toml
[output]
source = "{source_out}/{module}.{ext}"
test = "{test_out}/{module}.test.{ext}"
```

次のような出力になります。

| Markdown doc | 既定の出力先 |
| --- | --- |
| `.mds/source/greet.ts.md` | `src/greet.ts` |
| `.mds/test/greet.ts.md` | `tests/greet.test.ts` |
| `.mds/source/lib.rs.md` | `src/lib.rs` |
| `.mds/test/lib.rs.md` | override が無ければ `tests/lib.test.rs` |

## output override

一部 module だけ別の naming rule が必要なら `[[output.override]]` を使います。

```toml
[[output.override]]
match = "*"
kind = "test"
path = "{test_out}/test_{module}.{ext}"
```

minimal Python example ではこの形で `tests/test_{module}.py` を生成しています。

## safety rule

- generated output は package boundary の内側に限られます。
- mds は managed header が付いた file だけを上書きします。
- manifest が信頼できない場合は出力を書き込みません。

## managed file と manifest

generated file には次のような header が付きます。

```text
Generated by mds. Do not edit. Source: <path>. Source-Hash: <sha256>.
```

mds は `.mds/manifest.toml` に generation result を記録し、managed file と handwritten file を区別します。

## source map と editor bridge

generation planning では source map も記録されます。`mds-lsp` と VS Code 拡張はこれを使い、generated file の hover、definition、diagnostics を元の Markdown range へ remap します。

## 書き込む前の確認

まず dry-run を使います。

```bash
mds build --package ./path/to/package --dry-run
```

計画が正しければその後で書き込みます。
