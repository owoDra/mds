# 生成の仕組み

このページでは、mds が Markdown から派生コードを生成する規則を説明します。

## 生成元

生成元は、実装 Markdown の `Types`、`Source`、`Test` にあるコードブロックです。

mds は、説明文から実装を推測して生成しません。生成されるコードは、Markdown に書かれたコードブロックとメタ情報から決まります。

## 生成されるファイル

実装 Markdown の対象言語に応じて、生成先が決まります。

| 入力の例 | `Source` の例 | `Types` の例 | `Test` の例 |
| --- | --- | --- | --- |
| `src-md/foo/bar.ts.md` | `src/foo/bar.ts` | `src/foo/bar.types.ts` | `tests/foo/bar.test.ts` |
| `src-md/pkg/foo.py.md` | `src/pkg/foo.py` | `src/pkg/foo.pyi` | `tests/pkg/test_foo.py` |
| `src-md/foo/bar.rs.md` | `src/foo/bar.rs` | `src/foo/bar_types.rs` | `tests/foo_bar_test.rs` |

実際の出力先は、`mds.config.toml` の `roots` 設定と、言語アダプターの規則で決まります。

## コードブロックの連結

`Types`、`Source`、`Test` の中に複数のコードブロックがある場合、mds は出現順に連結します。

コードブロックの間には、説明文や補足見出しを置けます。説明文は生成コードには入りません。

## 依存宣言の生成

import、use、require などの依存宣言は、`Uses` の表から生成します。

依存の種類は、次の順序で扱います。

1. `builtin`
2. `package`
3. `workspace`
4. `internal`

同じ対象への名前付き依存は、可能な範囲でまとめられます。

## 生成ファイルのヘッダー

mds が生成するファイルには、管理ヘッダーが付きます。

ヘッダーには、生成元の Markdown と生成入力のハッシュが含まれます。

```text
Generated by mds. Do not edit. Source: <path>. Source-Hash: <sha256>.
```

このヘッダーにより、生成ファイルが mds の管理対象であることを判断できます。

## マニフェスト

mds は、パッケージ内の `.mds/manifest.toml` に生成結果を記録します。

マニフェストには、生成元、出力先、種別、ハッシュなどが入ります。

マニフェストが壊れている場合、mds は安全のため生成を書き込みません。

## 上書き規則

mds は、管理ヘッダーがある生成ファイルだけを上書きします。

管理ヘッダーがない既存ファイルが生成先にある場合、mds はエラーにします。これは、手書きのファイルを誤って破壊しないためです。

## 削除規則

生成元の Markdown がなくなった場合でも、mds は古い生成ファイルを自動削除しません。

古い生成ファイルは警告や確認の対象になります。削除は利用者が意図を確認してから行う前提です。

## 差分確認

生成前には、`mds build --dry-run` で差分を確認できます。

```bash
mds build --package path/to/package --dry-run
```

このコマンドはファイルを書き込まず、生成予定と差分を表示します。
