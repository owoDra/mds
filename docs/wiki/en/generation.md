# Generation Mechanism

> *This page was translated from [Japanese](../ja/generation.md) by AI.*

This page explains how mds turns authoring-v2 Markdown docs into package outputs.

## What Generates Output

- In source docs, executable output comes from `Source` fences.
- In test docs, executable output comes from `Test` fences.
- Prose, headings, and overview docs explain intent but do not generate code by themselves.

## Logical Module Ids

mds derives a logical module id from the Markdown path inside `.mds/source` or `.mds/test`.

Examples:

| Markdown doc | Logical module id |
| --- | --- |
| `.mds/source/greet.ts.md` | `greet` |
| `.mds/source/http/client.ts.md` | `http/client` |
| `.mds/test/greet.ts.md` | `greet` |

The module id is used in output patterns and wiki-style links.

## Default Output Planning

With the default config:

```toml
[output]
source = "{source_out}/{module}.{ext}"
test = "{test_out}/{module}.test.{ext}"
```

the following outputs are produced.

| Markdown doc | Default output |
| --- | --- |
| `.mds/source/greet.ts.md` | `src/greet.ts` |
| `.mds/test/greet.ts.md` | `tests/greet.test.ts` |
| `.mds/source/lib.rs.md` | `src/lib.rs` |
| `.mds/test/lib.rs.md` | `tests/lib.test.rs` unless overridden |

## Output Overrides

Use `[[output.override]]` when part of a package needs a different file naming rule.

```toml
[[output.override]]
match = "*"
kind = "test"
path = "{test_out}/test_{module}.{ext}"
```

This is how the minimal Python example writes `tests/test_{module}.py`.

## Safety Rules

- Generated outputs must stay inside the package boundary.
- mds only overwrites files that already have the managed header.
- If the manifest cannot be trusted, mds refuses to write output.

## Managed Files and Manifest

Generated files receive a header like the following.

```text
Generated by mds. Do not edit. Source: <path>. Source-Hash: <sha256>.
```

mds records generation results in `.mds/manifest.toml` so later runs can distinguish managed files from handwritten ones.

## Source Maps and Editor Bridge

Generation planning also records source maps. `mds-lsp` and the VS Code extension use them to remap hover, definition, and diagnostics from generated files back to the Markdown ranges that own the code fence.

## Before Writing Files

Use dry-run first.

```bash
mds build --package ./path/to/package --dry-run
```

Then write the outputs once the plan looks correct.
